When a decision is refined, edit its entry in place so it always describes the current choice. When a decision is fully superseded, delete the stale entry rather than layering "supersedes" notes on top of it. The log should read as what is true now, not as a history of what changed.

## Log

### 2026-05-12, shared-protocol, absolute coords on drag

Choice: `CDrag` and `SDrag` carry absolute `worldX, worldY` instead of deltas.
Why: robust to packet loss and reorderings, simpler client reconciliation, no integration of deltas. Cost is a few extra bytes per frame.
Revisit when: drag broadcast bandwidth shows up in profiling, or when we add per-frame rotation. Quantize to integer pixels or switch to int16 deltas if needed.

### 2026-05-12, shared-protocol, drop implies release

Choice: no explicit `release` message. `CDrop` finalizes the position and releases the cluster lock atomically.
Why: one round trip instead of two, one less message type, matches mouse-up semantics.
Revisit when: we need to distinguish "user let go" from "client disconnected mid-drag". Today both are handled by drop + connection close; if the snap/anchor logic later cares about that distinction, add an explicit release.

### 2026-05-12, shared-protocol, PROTOCOL_VERSION

Choice: `PROTOCOL_VERSION` is 3. The client sends it in `hello` and the server rejects a mismatch with `protocol_mismatch` (closing the socket) rather than letting a stale client hit malformed-message errors; `welcome` echoes the server's version. It is bumped on any breaking change to the wire union. v3 dropped the full-board `state` push on contributor connect: `welcome` carries no board, `region_state` was widened from positions to full construction data (origin, locked, size, pieceIds) so it both seeds the board per viewport and resyncs it, and a `minimap` server message carries the downsampled overview.
Why: a single asserted version turns a client/server skew into one clear error at the handshake. Bumping on every breaking surface change keeps the assertion meaningful.
Revisit when: any further breaking wire change bumps it again. Phase 2 freezes the protocol before public launch.

### 2026-05-20, shared-protocol, presence messages

Choice: presence is four message types. Client to server: `viewport` (visible world rect, for broadcast scoping) and `cursor` (pointer in world space). Server to client: `join` (userId, pseudo), `leave` (userId), and `cursor` (a peer's pointer, relayed to viewport-neighbor peers). There is no server `viewport` relay; cursor and viewport are split rather than carried in one combined presence message.
Why: viewport and cursor both change on every zoom and pan, but from independent triggers (wheel/pan vs pointermove), so a combined message would resend one half's unchanged fields on every tick of the other. Separate messages let each be throttled on its own cadence. Viewport has no client-facing consumer yet (only the server scopes broadcasts with it), so relaying it would be speculative. Cursor carries finite coordinates only: a pointer leaving the canvas is expressed by the client pausing its cursor stream, the same way `drag` stops at drop.
Revisit when: a minimap or overview wants to draw peer viewports (add a server `viewport` relay then), or the renderer needs to distinguish "pointer left the canvas" from "peer idle" (add an explicit off-canvas signal). Both `viewport` and `cursor` are now wired server-side: `viewport` is stored per client to scope drag and drop broadcasts, and `cursor` is relayed to viewport-neighbor peers. The pseudo carried on `join` is the authenticated profile pseudo, resolved at the WS upgrade and fixed for the connection (changing it requires reconnecting).

### 2026-05-12, piece-generation, canonical sign convention

Choice: each shared edge derives `sign` once from a canonical subseed. The piece whose `bottom` or `right` uses the edge takes the canonical sign, the piece whose `top` or `left` uses it takes the opposite. All other params (`center, neck, depth, shoulder, tension, tilt`) are shared as-is.
Why: all edges are traversed start-to-end in the same world direction by both neighbors (top/bottom left-to-right, left/right top-to-bottom). With identical traversal, only the sign needs to flip to express "bump out of A" vs "bump into B". Avoids mirroring continuous params.
Revisit when: never expected. If we change the traversal convention (e.g., for clockwise outline assembly in the renderer), this assumption must be re-derived.

### 2026-05-13, piece-generation, circular bulb head

Choice: edges carry 8 continuous params (was 6). Two new params added (`shoulderRun`, `headRoundness`) and existing params reinterpreted. The head is now a true circular bulb whose radius `r = headRoundness * depth` is strictly larger than the neck half-width by construction of the ranges, giving the classical lightbulb silhouette. Ranges: `center 0.46-0.54` (tab x position), `neck 0.055-0.085` (half-width of the neck pinch), `depth 0.24-0.30` (apex outward extent), `shoulder -0.025 to -0.005` (small undercut at the neck), `tension 0.25-0.40` (rise tangent scale), `tilt +-0.03` (bulb x asymmetry), `shoulderRun 0.10-0.16` (flat baseline length each side), `headRoundness 0.45-0.55` (bulb radius / depth ratio).
Why: earlier attempts that simply tweaked control-point offsets produced bulb widths comparable to the neck, which reads as a rectangular tab. The only robust fix is to parameterize the head as an actual circle whose radius is decoupled from the neck width, and enforce `r > neckHalfWidth` through the ranges (`r_min = 0.45 * 0.24 = 0.108 > 0.085 = neck_max`). Adding two floats keeps per-edge entropy comparable; the continuous space across ~2M edges remains non-degenerate.
Revisit when: silhouettes look degenerate, neighbors visibly misalign at snap, or we want more visual variety (widen ranges, vary bulb verticality, add a separate neck-height param). Validate against self-intersection once a large-N visual sample exists.

### 2026-05-13, piece-generation, edge path topology

Choice: the renderer (path.ts) walks each curved edge as 8 cubic Bezier segments: flat shoulder, rise-lower (baseline to neck pinch with undercut), rise-upper (neck pinch outward to bulb equator), bulb top-left quarter arc, bulb top-right quarter arc, fall-upper, fall-lower, flat shoulder. The two bulb arcs approximate a true circle using the canonical cubic Bezier handle length `0.5523 * r`.
Why: the bulb being a circle is what makes the silhouette read as a jigsaw piece rather than a generic curve. Separating the rise into "lower" (under the baseline, undercut) and "upper" (outward swing from neck to bulb equator) lets the neck pinch be an actual sharp narrowing rather than a smooth bulge. 4 or 6 segments do not have enough endpoints to encode a flat baseline + undercut + neck pinch + circular bulb.
Revisit when: GPU cost of triangulating ~8 cubics per curved edge at 1M pieces shows up in profiling. Drop the two flat-shoulder cubics in favor of `L` line commands first; further compaction would mean sacrificing the circular bulb.

### 2026-06-09, image-pipeline, pre-cut alpha masks baked server-side

Choice: the slicer bakes each piece's bezier silhouette into its AVIF alpha. Per piece it extracts the tile window (`pieceSize + 2 * margin`, centered, transparent-padded at the puzzle border) directly from the source file through libvips random access (no global RAM buffer), rasterizes the piece path from `@mpp/shared` to a white-on-transparent mask shifted by `+margin`, and composites it `dest-in` over the window before AVIF encoding. The same path is then stroked over the cut window in the shared `PIECE_BORDER_*` style (color/width/alpha live in `generator/path.ts`, one source for the bake and the client) and the manifest carries `borderBaked: true`, so the silhouette outline ships in the tile too. The manifest carries `premasked: true`; the frontend shows the tile as-is and applies no render-time silhouette mask or per-piece stroke (it still derives piece geometry from the seed for layout, the spatial index, the minimap and the snap-flash overlay), and the zoom-out LOD bakes pick up the baked border for free. The reference Deep Zoom pyramid is `dzsave`d straight from the source file. A procedural synthetic source (`scripts/synthetic-source.ts materialize`) writes a real tiled BigTIFF (flat, not pyramidal: 64-bit offsets for the >4 GB gigapixel raster, and the slicer builds its own reference pyramid), chunk-rendered to bound RAM, that the slicer treats exactly like a final photo, so the same code path runs for the gigapixel image by swapping the source file.
Why: the bezier params are frozen, so baking the mask once removes a per-sprite stencil mask from the render hot path (a Pixi `sprite.mask` breaks sprite batching and uses the stencil buffer per piece), the masking cost that did not scale to 1M; the inconvenient side (re-baking on a silhouette change) does not bite because the params never change at runtime. Reading per-piece windows from a tiled source keeps the slicer's peak RAM independent of source size, the requirement a gigapixel photo imposes. AVIF alpha is lossy, so the cut edge carries a few px of near-zero-alpha fringe inside the tile, imperceptible at piece scale and well within the margin (the silhouette clears the tile edge by ~2.8 px at the tightest, since `margin = round(0.35 * pieceSize)` exceeds the max tab depth `0.30 * pieceSize`).
Revisit when: the piece silhouette params or the border style (`PIECE_BORDER_*`) change (every tile must be re-baked and re-uploaded; the `--checksum` upload skips unchanged tiles, but a silhouette or border change touches every tile so it re-uploads the whole set), or rotation is enabled (the baked alpha fixes each tile's tab orientation, so a rotated render would need the mask reapplied live or per-rotation bakes).

### 2026-06-09, image-pipeline, synthetic source: banded assembly, deflate, ~6 GP machine ceiling

Choice: `materialize` assembles the source in bands, render each chunk at 1:1 to a tiled temp TIFF, composite a row of chunks into one full-width tiled strip, then stack the strips into the final tiled BigTIFF, with the libvips operation cache disabled so a finished composite releases its inputs (Windows otherwise refuses to unlink the still-open intermediates). Every intermediate is tiled so a region read costs one 512 px tile, keeping the live decoded set to a single band; a single composite over all chunks instead wants the whole raster decoded at once and overruns RAM. The codec is lossless deflate, not JPEG: libvips caps a JPEG-compressed image at 65500 px on an axis regardless of tiling, which a gigapixel width exceeds. On this Windows libvips build (sharp 0.34.5 / libvips 8.17.3) creating or writing any single image past ~6 GP segfaults (verified: 76000^2 = 5.78 GP writes, 80000^2 = 6.4 GP crashes; not RAM, since a blank black image crashes identically and neither tile size nor codec changes it). So the 1 000 000-piece synthetic test puzzle is built at piece-size 72 (72000^2 = 5.18 GP) to stay under the ceiling.
Why: piece count, not pixel size, drives every scaling concern the synthetic run exists to exercise (manifest size, 1M server init, Redis group count, region streaming); per-piece texel detail at 72 vs 80 px is irrelevant on a synthetic grid. Banding plus tiled intermediates keep peak RAM bounded by one band, the requirement a modest machine imposes.
Revisit when: processing the real 6-10 GP photo. It exceeds this build's ~6 GP ceiling no matter how the work is chunked (merging the parts into one source file, and the full-area reference `dzsave`, each need one image larger than the ceiling), so it must run on a 64-bit-clean libvips off this machine (the Linux VPS or a temporary larger box) and then `rclone` to R2.

### 2026-05-12, image-pipeline, tile margin

Choice: tile margin defaults to `round(0.35 * pieceSize)`, just above the max `depth` param (0.30) of the generator.
Why: ensures tabs always fit inside the tile with a small safety buffer.
Revisit when: edge param ranges widen, or rotation is enabled (tabs may then point in unexpected directions and the margin assumption changes).

### 2026-05-12, image-pipeline, adaptive pieceSize

Choice: `--piece-size` is optional. When omitted, the script derives `pieceSize = floor(min(width/cols, height/rows))` from the source image and center-crops the puzzle area to `cols*pieceSize` by `rows*pieceSize`. Any leftover band on the longer axis is discarded.
Why: the user wants to drop any image and have the pipeline adapt, not the other way around. Center-crop keeps the visually important center of the image.
Revisit when: a workflow needs to preserve the full image (no crop), align the puzzle to a non-center anchor, or use non-square pieces.

### 2026-05-12, backend-realtime, snap by origin equality

Choice: at drop time, the server collects grid-neighbor groups whose `worldX, worldY` is within `snapTolerance` of the dropped group's, skips any candidate currently held by another user, picks the target origin (a locked candidate when present, otherwise the first), then merges only the candidates also within `snapTolerance` of that target. No per-piece offset math.
Why: canonical offsets are puzzle-global (`col*pieceSize, row*pieceSize`), so two clusters are aligned exactly when their group origins are equal, which keeps snap detection to coordinate comparisons. The target-filter pass is required because two candidates can each be within tolerance of the dropped group while being up to `2 * snapTolerance` apart from each other and from the target the merge snaps everything onto. The held-candidate skip is a deliberate ownership rule, not the sub-millisecond discovery race in the per-group dispatch entry: a held cluster belongs to its holder and its stored position is frozen at grab time (drag never persists), so absorbing it would yank it out of the holder's hand at a stale position. The skip keys on `heldBy`, which `readGroup` parses to null for both free and locked groups (only an actively-held group carries a userId), so anchor-by-merge targets still merge; the join is not lost because the holder's own drop re-runs detection. The skip closes the deterministic case (the cluster is held throughout the drop); a grab landing between the candidate read and the merge's lock acquisition is the separate sub-millisecond window already covered by the per-group dispatch entry.
Revisit when: rotation is enabled (origins no longer suffice; will need per-edge alignment) or canonical offsets stop being puzzle-global.

### 2026-05-12, backend-realtime, docker all workspace deps

Choice: the server Docker image runs `npm ci --omit=dev` at the repo root, installing runtime deps for every workspace including the frontend, rather than scoping to `@mpp/server` and `@mpp/shared`.
Why: monorepo workspace filtering with `npm ci` is brittle (lockfile drift, missing peer resolution). Installing everything is one line, deterministic, and the image still excludes dev deps.
Revisit when: image size or cold-start time matters (Phase 1+ deploys). Switch to a server-only install or split lockfiles per workspace.

### 2026-05-12, shared-protocol, piece geometry not on the wire

Choice: piece silhouettes and canonical offsets are recomputed from `generationSeed` on both sides, never serialized.
Why: at 1M pieces, geometry would dominate payload size. Seed-based determinism keeps state minimal and timelapse replay tractable.
Revisit when: never expected. If the generator becomes non-deterministic across platforms (FP drift), pin to a fixed integer-math implementation rather than start shipping geometry.

### 2026-05-12, frontend-canvas, bounding rect hits

Choice: each piece's interactive area is the Pixi container default (children bounds), which is the sprite's full bounding rect (tileSize square), not the visible silhouette, whether the tile is a premasked alpha-cut AVIF or masked at render time (a sprite's bounds are its dimensions regardless of alpha or an applied mask). In overlap zones between adjacent unmerged pieces, clicks may pick either piece; topmost (z-order) wins.
Why: silhouette-shaped hit testing requires a custom `hitArea.contains` walking the cubic Bezier polygon. The bounding-rect default ships immediately and is good enough on a freshly shuffled board where pieces rarely overlap visibly.
Revisit when: overlap zones produce confusing pickups in practice, or once pieces start clustering. A `Polygon` hitArea sampled from the silhouette path is the natural follow-up.

### 2026-05-12, frontend-canvas, drag no throttle

Choice: every `pointermove` while a piece is held emits a `drag` WS message. No coalescing.
Why: simplest correct behavior. At Phase 0 (one client, low-rate mice) bandwidth is irrelevant, and the server already serializes messages.
Revisit when: high-rate mice (240Hz+) or many concurrent draggers create WS backpressure. Coalesce to one message per `requestAnimationFrame` tick and send the last point.

### 2026-05-12, frontend-canvas, cascade deferred

Choice: the cascade entrance animation is removed from Phase 0 and moved to Phase 2. The Phase 2 frontend-canvas entry describes it as a synchronized event-start cinematic, with pre-reqs in shared-protocol (`eventStartsAt` in welcome) and frontend-shell (landing countdown).
Why: the product vision is an event with a countdown on the landing page: at t=0 the canvas opens and every client plays the same cascade simultaneously, conveying scale and hype. Late joiners skip it. None of that is testable in Phase 0 (no event scheduling, no countdown, no synchronized trigger), and a local "play on /play open" cascade would be rewritten when the real trigger lands. Phase 0 ships without cascade; the other animations (snap, end-of-puzzle) still carry the "feel alive" pillar.
Revisit when: starting Phase 2, or earlier if the Phase 1 alpha would benefit from a dry-run of the event-start ritual. Implementation will be a one-shot Pixi ticker animation triggered when `Date.now() >= eventStartsAt - bufferMs` and the client has just connected.

### 2026-05-29, backend-realtime, per-group dispatch queues

Choice: WS messages are dispatched on per-group queues (`GroupQueue` in `queue.ts`), keyed by group id. Tasks whose key sets are disjoint run concurrently; tasks sharing any group id run in submission order, so a group's Redis read-modify-write never interleaves with another task touching the same group. Routing in `dispatch`: `grab` and `drag` run on `[groupId]`; a `drop` runs on the dropped group, and when its snap reaches neighbours it does not yet hold, `handleDrop` returns the full set and the drop re-runs holding the union (the merge mutates nothing until it holds every group it joins); `dev_reset`, `dev_complete`, and `dev_place` take a global barrier (`runGlobal`) because they read or rewrite the whole board; `hello` (no longer a board read under protocol v3), `viewport` and `cursor` run inline (no group state); disconnect release reads the connection's tracked held group ids (`Client.held`, added on a winning grab and removed on the drop that ends the hold) and releases them on their queues with a held-by re-check, so it is O(held) and never scans the board. Keys are acquired synchronously at submission, so the dependency graph follows submission order and cannot deadlock even when a task spans several groups (two drops merging the same pair from opposite sides converge on one shared lock set; the loser re-reads and finds its group already merged away).
Why: one global serial queue made every message wait behind every other, so independent groups across the canvas could not progress in parallel; per-group queues keep the per-group ordering the handlers depend on while letting disjoint work run concurrently. The global locked-count stays correct without a global lock because anchoring increments it with an atomic Redis `INCRBY` (concurrent anchoring merges on disjoint groups cannot lose an update, and exactly one crosses the completion threshold).
Revisit when: the writer path needs more than one instance. The guarantee is per process: `GroupQueue`'s key tails live in process memory, so horizontally scaling the WS writer breaks it, and the options are an atomic Lua merge, a distributed lock per canvas region, or write sharding by region. One known per-process cost: a merge cannot reserve a neighbour's lock until it discovers the neighbour, so a neighbour op landing in the sub-millisecond discovery window may interleave (re-validation keeps state correct, with the same net effect as the old submission-order race).

### 2026-05-18, frontend-shell, dev controls

Choice: `/play` exposes three buttons (Place piece, Reset puzzle, Complete) wired to `dev_place`, `dev_reset` and `dev_complete` WebSocket messages. `dev_reset` wipes and re-initializes the current puzzle (and clears its merge log, so the leaderboard and activity feed start empty); `dev_complete` jumps the locked count to the total, flips the meta status to `completed`, and logs one merge crediting the executing client for every piece not already attributed; `dev_place` anchors one random cluster. Visibility is controlled by `VITE_DEV_BUTTONS` (default visible during the alpha); within that, the controls show in any session mode, not just contributor. They send over the WebSocket, so clicking one in spectator mode upgrades the session to a contributor connection and the message is queued and flushed on `welcome`. Server-side every dev message is rejected with `dev_disabled` unless `MPP_DEV_ENABLED=1`. Reset and Complete are protected by a `confirm()` prompt because they affect every connected tester at once.
Why: testers need a way to exercise the end-of-puzzle UI and to reset a stuck board without operator intervention. Putting the controls on `/play` (rather than a hidden URL) keeps feedback loops short. The env gate exists so we can pull them in one redeploy when the alpha ends.
Revisit when: the alpha ends. Flip `MPP_DEV_ENABLED=0` on the server and `VITE_DEV_BUTTONS=0` on the frontend before any non-tester traffic lands.

### 2026-05-18, backend-realtime, WS hardening

Choice: the WebSocket server enforces five limits at the network boundary. (1) `verifyClient` rejects upgrades whose `Origin` header is not in `MPP_ALLOWED_ORIGINS` (comma-separated; default `*` with a boot warning). (2) `maxPayload` caps a single frame at `MPP_WS_MAX_PAYLOAD_BYTES` (default 64 KB), so `ws` rejects oversize frames before they reach `JSON.parse`. (3) A per-IP `TokenBucket` (capacity `MPP_WS_RATE_BURST`, refill `MPP_WS_RATE_TOKENS_PER_SEC`, defaults 400 / 200 per sec) is consumed once per inbound message; over-budget messages are dropped silently before the dispatch queue sees them. (4) An `IpRegistry` caps concurrent connections per IP at `MPP_WS_MAX_CONNECTIONS_PER_IP` (default 10); a connection over the cap is closed with code 1013 before it is added to the hub. (5) `Hub.send` and `Hub.broadcast` close the connection with code 1013 ("Try Again Later") when `ws.bufferedAmount` exceeds `MPP_WS_BUFFERED_AMOUNT_LIMIT_BYTES` (default 4 MB), so a slow consumer cannot grow the writer's memory without bound. The rate bucket and connection count are keyed by client IP and shared across all of an IP's connections through the `IpRegistry`, whose entry is created on the first connection and deleted when the last one closes. The IP is read from the `CF-Connecting-IP` header set by Cloudflare (`socket.remoteAddress` is the edge, not the client), falling back to `socket.remoteAddress` in dev and to a shared `unknown` key for a production request that did not arrive through the edge.
Why: without these, a single client can CSRF-connect from any origin, send a 100 MB frame, flood drag messages at arbitrary rate, open unbounded parallel sessions to multiply its rate budget, or stall on socket reads while the writer queues snap broadcasts forever. None of these are theoretical: a few lines of JS from any tab were enough before. Keying the rate bucket and connection cap by IP rather than by connection means opening more sessions cannot raise an IP's aggregate message rate or socket count, which a per-connection bucket alone allowed. Putting the checks at the boundary keeps the handler code unchanged and the budgets all live in `config.ts`. Silent drop on rate overflow (no error frame) avoids amplifying a hostile client's traffic.
Revisit when: the load tests in `qa-and-load` run. Tune the burst/rate and the connection cap against measured legitimate usage (240Hz mice during a multi-piece cluster drag; testers behind one shared NAT count against a single IP budget, so the cap may need raising). Trusting `CF-Connecting-IP` assumes ingress is restricted to Cloudflare: a direct-to-origin client can spoof the header, so the deployment must firewall the origin to Cloudflare ranges (the alpha leaves this open). The per-IP key is per address, so an IPv6 client with a wide prefix can rotate addresses to evade the cap; bucket by `/64` if that shows up. The `IpRegistry` lives in one process, so sharding the WS writer needs the count moved to shared state (e.g. Redis). The Origin allowlist stays in place; the upgrade now also requires a valid session cookie (see the WS session gate), so anonymous connections are rejected regardless of Origin.

### 2026-05-18, infra-deploy, alpha topology

### 2026-06-06, backend-realtime, spatial broadcast index and cluster-AABB scoping

Choice: drag, drop and cursor broadcasts are scoped through a spatial index instead of a linear scan of every connected client. The Hub keeps a `Map<cellKey, Set<Client>>` over the shared world grid (cell = `WORLD_TILE_SIZE`, the one grid the frontend bakes LOD tiles on, the group index files positions on, and the per-tile piece cap counts over, so a zoomed-in viewport overlaps ~1-4 cells) plus a global subscriber set for fail-open. A `viewport` message diffs the client's old and new overlapped cell set and updates its subscription in O(cells); a viewport overlapping more than `MPP_BROADCAST_MAX_CELLS` (default 256) cells, or none yet, makes the client a global subscriber, bounding the per-client subscription set. A scoped broadcast walks only the cells the event AABB overlaps plus the global set, so its cost tracks subscribers in range, not connected-client count (the client sits in cells XOR the global set, deduped per broadcast). Scoping is by the dragged cluster's world axis-aligned bounding box, not its origin point: each group stores its group-local AABB (`aabbMin/Max X/Y` on the Redis group hash), computed once at the initial write and at every merge from the member piece ids with the same `col = id % gridCols, row = floor(id / gridCols)` arithmetic `detectSnap` uses (a singleton is one piece footprint, a cluster the union). The world AABB is the local AABB translated by the live origin, so the drag hot path reads only the group (no per-frame piece scan). Snap, join, leave, grab_ok, leaderboard and completion stay global broadcasts. The local AABB never crosses the wire (`readAllGroups`, feeding `state` and the keyframe, returns plain `GroupRuntime`); a group written before AABBs were stored reads a null local AABB and falls back to point scoping until its next merge or a reset re-init.
Why: the linear scan was O(connected clients) per drag, cursor and drop, the bottleneck at public scale on a 1M canvas; the spatial index makes a scoped broadcast independent of how many clients are connected. Storing the local AABB once at merge (not recomputing a cluster bbox per drag frame, which an earlier point-scoped design avoided as too heavy) is what lets scoping use the full cluster extent with no per-frame piece read: it fixes the gap where a large cluster whose origin sat off a peer's screen while its body overlapped it was missed (a single non-merging drop never self-corrected). The cell cap trades exact scoping for a bounded subscription set: a far-zoomed viewport receives every scoped event as a global subscriber rather than subscribing to thousands of cells, acceptable because WS clients are active solvers (typically zoomed in to place pieces) while passive far-zoomed viewers read the spectator HTTP stream, not a socket. The four AABB numbers per group are the Redis cost paid for the per-frame-read-free hot path.
Revisit when: the writer is sharded (the index lives in process memory, like the dispatch queues and the IP registry; a second writer needs the subscription state in shared state). The read-path counterpart, a peer panning into a region still holding stale positions for non-merging drops it missed, is handled by the [group index + pan resync](#2026-06-06-backend-realtime-group-spatial-index-and-pan-resync) below.

### 2026-06-06, backend-realtime, group spatial index and pan resync

Choice: a peer panning into a region is resynced to the current positions of the groups there, closing the read-path gap the broadcast index left open (a non-merging drop is a scoped broadcast and is not persisted, so a client that was looking elsewhere never received it and holds a stale position; snaps are global, so only non-merging drops can go stale). The server keeps a second in-process spatial index, `GroupIndex`, on the same world grid as the broadcast index: a `Map<cellKey, Set<groupId>>` plus a reverse `Map<groupId, {cell, originX, originY, size, locked}>`. Each group is filed in exactly one cell, the cell of its body top-left (its world AABB min, where the client sees it), while the stored payload is the group ORIGIN a canonical offset away (what the client positions the container at) plus its size and locked state. It is maintained on every committed change: a non-merging drop re-keys the group, a merge removes the merged-away groups and re-keys the survivor, and init/reset/force-complete rebuild it. Storing origin/size/locked makes the gather a pure in-memory read (no per-pan Redis scan, like the broadcast index); only a size>1 group needs a Redis read, for its member piece ids. When `handleViewport` diffs the client's overlapped cells, it gathers the groups in the newly entered cells (cells the client already held are not re-sent), attaches each group's piece ids (the singleton's own id, or a Redis read for the few size>1 groups), and sends a `region_state` construction stream. The client upserts each entry: it builds an unknown group wholesale (the per-viewport board seed under protocol v3, where welcome carries no board), and for a known group applies the origin behind an ordering guard (`resyncShouldApply`): it skips a group it is holding, a peer is holding (`heldGroupIds`, known from the global `grab_ok`), or one it just dropped and is awaiting confirmation of (`pendingDrops`), so a resync (which carries the last committed resting position) never rewinds a newer live update the client is the authority for; every other group takes the origin and additively reconciles membership/locked, and sequential application of any later live drag/drop/snap stays correct because the connection is ordered and the index is monotonic. This index serves both the viewport-scoped initial state and the incremental-subscriptions read path.
Why: the broadcast index made live drops reach a region's current viewers, but did nothing for a client arriving later, the gap that single non-persisted drop could not self-heal. A per-group index keyed on the broadcast grid answers "which groups sit in these cells" in O(cells) the way the broadcast index answers "which clients overlap this event". Single-cell-per-group (body min, not the multi-cell AABB the broadcast scopes with) keeps it one entry per group, ~1M at the start (each piece its own group) and shrinking with every merge, and keeps a giant assembled cluster from occupying thousands of cells; the cost is that a cluster straddling a cell boundary is resynced only via its min-corner cell, healed by its next live drop or a continued pan. Storing positions in the index duplicates Redis but keeps the throttled pan path off Redis, the same read-model trade the broadcast index makes for subscriptions. The index is process-memory (single-writer), rebuilt from Redis at boot and reset because Redis survives a restart and the index does not. The resync deliberately re-sends every group in the entered cells, including unchanged singletons the client already has at their scatter position (idempotent, and the client skips a no-op move), bounded by the broadcast cell cap (a viewport overlapping more than `broadcastMaxCells` is a global subscriber and gets no per-cell resync); sending only drop-moved groups would shrink the payload but adds a dirty-set to maintain, deferred until the payload bites.
Revisit when: the writer is sharded (this index, like the broadcast index and the dispatch queues, is per process and would need shared state). If the per-cell payload (full cells, not just moved groups) shows up, track a moved-since-init set and stream only those. See [viewport-scoped initial state on join](#2026-06-06-backend-realtime-viewport-scoped-initial-state-on-join) for the join semantics this index serves.

### 2026-05-21, frontend-canvas, leaderboard scoring

Choice: the leaderboard scores each piece one point, credited to the user of the first `ClusterMerge` (by `at`) whose `droppedPieceIds` lists it. `ClusterMerge` stores `droppedPieceIds`, the pieces of the group the user dragged in that merge, alongside `addedPieceIds` (the pieces whose group id changed, kept for client sprite re-parenting). Standings are derived on demand by a `cluster_merges` aggregation, broadcast after every anchoring snap and sent to each client on join, so the in-game leaderboard panel and the completion modal both stay live. Rows display the contributor's pseudo, resolved from the user profile by a `$lookup` into `users`, falling back to a shortened `userId` when unset.
Why: a piece moves only when its group is dragged, and every piece starts shuffled and must be carried to its solved position (a misplaced cluster cannot be bridged into the solved structure, since a snap needs both sides already aligned within tolerance). So every piece is in some merge's dragged group at least once, and crediting the first such merge gives each piece exactly one point, with per-user totals summing to the full piece count. `droppedPieceIds` is the correct basis: `addedPieceIds` records the lower-group-id side of a merge, not the side the user dragged, so it would credit the stationary cluster when a low-id cluster is dragged onto a high-id target. The pseudo lives on the Mongo user profile, so the aggregation attaches it with a final `$lookup` into `users`, tolerant of ids that are not valid ObjectIds (dev data) via `$convert ... onError: null`.
Revisit when: the aggregation unwinds and groups the full merge log; re-running it on every anchoring snap will not scale to a 1M-piece puzzle, so precompute a per-user counter then.

### 2026-05-20, shared-protocol, lockedDelta stored on ClusterMerge

Choice: `ClusterMerge` persists `lockedDelta`, the number of pieces a merge newly locked to the frame, even though stats are otherwise derived on demand.
Why: the activity feed backfill sends a per-merge "placed N pieces" count for anchoring events, and that count cannot be reconstructed from a saved merge. A frame-anchored cluster locks its pieces without listing any in `addedPieceIds` (those ids only ever hold pieces that changed group id), so the count is known only at merge time. It is a property of the event, not a precomputed per-user counter, so it does not conflict with deriving user stats on demand.

### 2026-05-21, frontend-canvas, frustum culling and tiled zoom-out LOD

Choice: three techniques keep rendering cheap, all bounded by the visible window through one shared spatial index. (1) Spatial index: a `GroupGrid` (`groupGrid.ts`, Pixi-free, unit tested) keyed on a fixed world grid whose cell equals the LOD tile size, mapping cell -> groups and group -> cells. Maintained at `build()`, on every `moveGroup`, and at every group removal, in O(cells per group) and off the dirty path. (2) Frustum culling: `cullAll` asks the index for the groups near the viewport and runs the precise per-group test on those candidates only (O(visible), not O(board)); groups that left the previous candidate set are culled via Pixi's `culled` flag. The test is two-level: a group whose world AABB misses the viewport is culled whole; one that intersects has each piece tested individually. Bounds are analytic (canonical offset plus one margin per piece, unioned per group), never `getBounds`. (3) Tiled zoom-out LOD (`lodTiles.ts`): below `LOD_ENTER_ZOOM` (0.3) the play zone is shown through a grid of small render-texture tiles instead of the per-piece masked sprites; above `LOD_EXIT_ZOOM` (0.35) the live pieces return (the gap is hysteresis). Each tile is `LOD_TILE_WORLD` (2048) world px square at `LOD_TILE_DENSITY` (0.22) device texels per world px (dpr folded into the texel count), so per-piece sharpness is constant regardless of board size and device, ~1.5x sharper than a screen pixel at `MIN_ZOOM` (vs ~0.05 texel/world-px for the prior single play-zone texture on the alpha board). Tiles are baked on demand from the live scene, cached, and the resident set is held to an LRU cap of `max(LOD_VRAM_BUDGET_MB tiles, screen-cover tiles)`: the screen-cover floor (tiles covering the `MIN_ZOOM` viewport plus a margin ring) guarantees a needed tile is never evicted (no re-bake thrash), and a hard `LOD_VRAM_CEILING_MB` (768) degrades density on extreme displays rather than exceeding VRAM (density is the only VRAM-effective knob, since shrinking the tile world size keeps the total screen-cover texel count constant). A bake queue drains `LOD_BAKE_PER_FRAME` (2) tiles per frame; in the warm band (`LOD_ENTER_ZOOM .. LOD_WARM_ZOOM`, 0.5) it pre-bakes the visible tiles while the live pieces still show, so a progressive zoom-out into the active band does not hitch. A direct jump below `LOD_ENTER_ZOOM` (cold start, fit button, double-click, keyboard) has no warm interval, so `setLodActive` bakes the screen-cover tiles in one synchronous burst on entry rather than letting the 2/frame queue render every visible cluster live for ~0.3-1s. Each tile bakes only its cell's clusters (queried from the index), clipped to the tile, with held clusters, the frame, the backdrop and the tile layer excluded; the frame and backdrop stay live crisp overlays. Gapless fill: while the LOD is active a non-held cluster renders live until every tile it occupies is baked, then hides (the tiles draw it); held clusters always render live on top. Dirtying is per tile and covers both the rect a cluster leaves and the one it enters (a grab excludes a cluster from bakes though its position is unchanged, so its resting tile is dirtied on the grab transition; a drop, snap, rollback or grab-denied dirties the old and new tiles); a spectator keyframe re-base dirties the tiles of the clusters it changes, and a spectator drop glide dirties the gliding cluster's tiles each frame so it renders live until it settles. The per-frame drag of an already-held cluster stays off the dirty path. Interaction follows visibility: a resting cluster hidden behind its tiles is non-interactive (Pixi does not hit-test an invisible container, and the tile layer is `eventMode = "none"`), so below `LOD_ENTER_ZOOM` no new grab can be initiated and the canvas only pans; a cluster already in hand stays live and keeps dragging (held clusters are excluded from tiles), so only starting a grab is blocked.
Why: culling removes off-screen per-piece stencil and draw-call cost, but does nothing fully zoomed out where every piece is visible and every mask still renders. The single play-zone render texture it replaced did not scale: per-piece sharpness collapsed as the board grew (one piece mapped to ~3 texels on the alpha board, far less at 1M) and a full-world re-bake was O(board). The tile grid makes per-piece density constant and bounds VRAM and per-frame bake cost to the visible window; the folded index makes both the live cull and the per-tile bake O(visible) rather than O(board). Baking on the client (not from the libvips pipeline) is required because the pipeline is offline and only late-solved regions match the source image, so the shuffled board is live state.
Revisit when: Phase 2 at 1M pieces. The runtime paths (cull, per-tile bake, pan, gapless fill, VRAM) are already bounded by the visible window, and the one-time `build()` (one container per group plus the index population) now runs in yielding chunks (see [chunked board build](#2026-06-04-frontend-canvas-chunked-board-build-with-lazy-geometry)), so it no longer freezes the main thread; full 1M runtime verification is folded into "Rendering stays smooth on commodity hardware at 1M". The LOD is verified at the alpha's 10 000. Separately, the active band is effectively overview-only (no grab below `LOD_ENTER_ZOOM`, where a piece is ~24 px at 0.3 and ~12 px at `MIN_ZOOM`); accepted for now. If contributors need to grab at far zoom, keep the resting clusters hit-testable while hidden (render with `renderable = false` instead of `visible = false`, or overlay a transparent per-cell hit layer) rather than lowering `LOD_ENTER_ZOOM`.

### 2026-05-21, frontend-canvas, play-zone hard limits

Choice: the board is bounded by a play zone, computed once on the server (`playZoneForManifest` in `init.ts`) as the AABB of the puzzle frame unioned with every piece at its initial scattered position, then widened by a margin (50% of its larger side), mirrored around the frame center, and with the symmetric half-extent snapped outward to the world grid cell (`GRID_WORLD_CELL`, 80, in `@mpp/shared`). The server sends the zone in the `welcome` message, so every client of a puzzle enforces the exact same bound regardless of join time. The camera is clamped to the play zone expanded by a padding ring (4% of its larger side); when the viewport is larger than that limit on an axis it centers instead. Held pieces are clamped at drag and drop input so no piece can leave the zone. The area outside is darkened by a world-space fill carrying a coarse checkerboard, a distinct motif from the hairline grid inside. The minimap canvas adopts the zone's aspect ratio (clamped) and insets the zone within a thin out-of-bounds band, so the map fills the panel with no letterbox and the outside reads on every edge. Minimum zoom is raised from 5% to 15%.
Why: computing the zone on the server makes it a single authoritative value. A per-client computation from join-time positions would give late joiners a smaller zone, since pieces drift inward as the puzzle is solved: a piece dragged to an early client's zone edge would land outside a late client's bound, unreachable in the dark backdrop, with camera limits and minimap extent diverging between clients. The zone is a pure function of the manifest seed, so the server computes it once at boot in `PuzzleLifecycle` rather than storing it. The margin gives pieces scattered against the raw bound room to be dragged outward; the zone still never needs to grow, since the clamp keeps every piece inside the padded bound (a drag is clamped, a merge lands within existing pieces). Mirroring the bound around the frame center keeps the puzzle frame dead-center in the minimap and gives the camera symmetric pan limits; snapping the symmetric half-extent to the grid cell lands the backdrop boundary on a grid line when the frame center is itself grid-aligned (the 1M board). The piece clamp is client-side only, since the WS server does not validate positions, consistent with the closed alpha's honest-client assumption. The 15% zoom floor bounds the visible piece count, so a large board no longer fits entirely on screen.
Revisit when: a malicious client sends out-of-zone positions (move the clamp server-side then), or Phase 2 viewport sharding changes how the world is bounded.

### 2026-06-05, backend-realtime, spectator keyframe and event log

Choice: spectator mode is a keyframe plus an ordered event log, both served over HTTP from the WS host and CDN-fronted. `GET /keyframe` serves an in-memory full-board body (`pieces`, `groups`, standings, plus the stream parameters `windowMs`, `delayMs`, the build `cursor`, and a `live` flag); `KeyframePublisher` regenerates it at most every `MPP_KEYFRAME_INTERVAL_MS` (300000) and only while the event is live (`status === "active"` and past `eventStartsAt`), freezing on the last body before the start and after completion so it does zero full-board reads when idle (a reset/complete transition forces one regeneration; the `Cache-Control` is `max-age=<interval>` when live and a short `max-age=<MPP_KEYFRAME_IDLE_TTL_MS/1000>` when frozen so the transition is seen promptly). `GET /events/<t0>` serves an immutable wall-clock window `[t0, t0+W)` of drops and snaps (`MPP_EVENT_WINDOW_MS`, 3000), addressed by `t0 = floor(t/W)*W`: a non-W-aligned `t0` is 400, a not-yet-sealed window (`now < t0+W`) is 425, an out-of-retention window is 404, all `no-store` and rejected before any Redis read; a sealed in-range window is `public, max-age=31536000, immutable`. Events live in a Redis stream (`puzzle:{id}:events`); the entry id (`<ms>-<n>`) is the event `seq`, its ms component is the window key, and the log is trimmed to `MPP_EVENT_RETENTION_MS` (900000). The client loads the keyframe, renders `MPP_INTERP_DELAY_MS` (6000) behind live, tails windows in order (each fetched once), interpolates a group's origin between drops, replays snaps in `seq` order, and re-bases onto a freshly fetched keyframe at the render time it represents (no jump). The stream carries its own `SPECTATOR_FORMAT_VERSION`, independent of the WS `PROTOCOL_VERSION`; a mismatch errors the client. A transient Redis hiccup keeps the last good keyframe body served rather than producing a 5xx.
Why: a full snapshot per tick is O(board) (~25 MB gzipped per tick at 1M), which does not scale; a keyframe plus an event log makes the per-tick payload proportional to the events in the interval, independent of piece count. Immutable wall-clock windows (no `?since` cursor) keep every window a single CDN-cacheable body shared by all clients, at the cost of a fixed `W` granularity and the keyframe being the only personalized fetch. The idle gate removes the full-board read entirely before the event and after completion, where the board never changes. A Redis stream (over a list or pub/sub) survives a restart, gives monotonic ids usable as both the dedup cursor and the window key, and is ready for a future reader-split. The client interpolation trades a few seconds of intentional delay for a small payload and an alive rendering (entity interpolation from multiplayer netcode): a drop is shown as a straight glide from the previous resting position, not the real drag path, and the spectator leaderboard refreshes only per keyframe (every 5 min) while the locked count and activity ticker stay live from snap events.
Revisit when: the fixed `W` granularity or the missing `?since` cursor forces too-coarse or too-frequent window fetches (add a cursor and accept per-client window bodies, or a second WS reader tier), or the 5-minute spectator leaderboard refresh is too stale (push standings on a lighter channel). The keyframe still reads the board off the write queue (see the off-queue read entry).

### 2026-06-07, backend-realtime, spectator stream anti-abuse

Choice: the anonymous spectator HTTP stream (`GET /keyframe` + `GET /events/*`) is guarded at the origin by `makeSpectatorGuard` (in `httpApp.ts`), registered before the stream handlers. A preflight `OPTIONS` passes straight through (the handler answers 204 with wildcard CORS). Every other request is first counted against a per-IP Redis fixed window (`RedisFixedWindow`, bucket `spectator`, key `ratelimit:spectator:<ip>`, default 120 requests / 60 s via `MPP_SPECTATOR_RATE_MAX` and `MPP_SPECTATOR_RATE_WINDOW_SEC`, fail-open on a Redis error); over budget is a 429 with `Retry-After: 1`. A survivor carrying any query string is then rejected with a 400. Both rejections carry `Access-Control-Allow-Origin: *` and `Cache-Control: no-store`. The IP is `CF-Connecting-IP` (the spectator host is Cloudflare-proxied), falling back as in the WS path. This reuses the same `RedisFixedWindow` as the auth routes, so `keys.authRate` was renamed `keys.rateLimit` now that two buckets share it.
Why: going public exposed the spectator read path as the only unauthenticated, previously unlimited endpoint. Most legitimate reads hit the CDN edge (the keyframe carries `max-age`, sealed windows are `immutable`), so the origin sees only cache misses and the per-IP window can sit far above a real spectator's origin rate (tens/min) yet still trip on a flood. The query-string rejection closes a cache-bust amplification: the handlers strip the query for routing, so without the guard `?cb=N` returned a fresh cacheable 200 per key, each a distinct edge cache entry and (for events) a fresh Redis `readWindow`; the client never appends a query, so rejecting it costs nothing. Rate-limit runs before the query check so every request class (including cheap cache-bust 400s) is capped per IP. Reusing the Redis fixed window over an in-process bucket keeps the counter correct if the WS writer is later sharded and shares the auth limiter's code. Wildcard CORS + `no-store` on rejections matches the stream handlers' conventions and stops the edge caching an error body.
Revisit when: volumetric DDoS spread across many IPs is Cloudflare's layer (the `infra-deploy` DDoS posture), not this per-IP origin window; tune the budget against the `qa-and-load` measurements; bucket by IPv6 `/64` if address rotation evades the per-IP key (the same caveat the [WS hardening](#2026-05-18-backend-realtime-ws-hardening) carries). If the Cloudflare Cache Rule is set to drop the query string from the cache key, the origin query rejection becomes belt-and-suspenders; keep it, since it is infra-independent. Per-account gameplay limits remain unbuilt.

### 2026-05-23, infra-deploy, snapshot proxied hostname

Choice: spectator traffic uses a dedicated hostname `snapshot.millionpiecepuzzle.com`, proxied through Cloudflare (orange cloud), pointing at the same Coolify service as the WS host. A Cloudflare Cache Rule scoped to that host and the spectator paths (`/keyframe` and `/events/*`) sets cache eligibility to "Eligible for cache" and edge TTL to "Respect origin", and enables "Serve stale content while updating" so a regeneration overrun never returns origin errors to viewers; the keyframe respects its short or long `max-age` and the sealed event windows send `immutable`, so the edge holds them long-term. The Node `/keyframe` and `/events/<t0>` handlers hardcode `Access-Control-Allow-Origin: *`, independent of the WS `MPP_ALLOWED_ORIGINS` allowlist which stays strict. The origin also guards these paths with a per-IP rate limit and a query-string rejection (see [spectator stream anti-abuse](#2026-06-07-backend-realtime-spectator-stream-anti-abuse)), complementing the edge cache.
Why: a dedicated hostname isolates the spectator cert and cache lifecycle from the WS path, so the spectator Cache Rule never has to be scoped narrowly enough to exclude the live socket on a shared host. A path-scoped Cache Rule keeps any future endpoints under the same host (`/healthz`, future R2-fronted assets) uncached by default. CORS on the spectator paths is wildcard because the payload is anonymous read-only puzzle state intentionally fronted by a shared CDN: a per-origin echo with `Vary: Origin` would cache-poison on Cloudflare Free (which does not honor Vary for caching), and an `Origin` header is trivially spoofed outside browsers so it is not a meaningful security boundary for an HTTP read endpoint. The WS Origin allowlist is a separate concern (anti-CSWSH from third-party tabs) and stays specific.
Revisit when: `ws.*` is now Cloudflare-proxied too (see the alpha-topology entry), so the dedicated hostname is no longer needed to keep WS off the proxy; folding spectator traffic back onto the WS host and retiring `snapshot.*` is available but deferred, kept for now because the path-scoped isolation is cheap and a Cache Rule on the WS host is easy to mis-scope. Retire it when that simplification is worth a coordinated `VITE_SPECTATOR_BASE_URL` change.

### 2026-05-23, image-pipeline, piece tiles bucketed by hundreds

Choice: the slicer writes piece tiles to `pieces/<bucket>/<id>.avif`, where `bucket = floor(id / 100)` zero-padded to 4 digits and `id` zero-padded to the width of the largest piece id. Each entry's `file` in the manifest carries the bucketed path verbatim; the frontend never reconstructs it from `id`. The slicer is the only path that emits AVIF, so there is no flat layout to maintain in parallel.
Why: at 1M pieces, a single flat `pieces/` directory holds a million entries, which slows `ls`, R2 listings, and any human inspection of the layout. Splitting by hundreds gives ~10 000 directories with up to 100 files each at full scale, both numbers comfortable to enumerate. Bucket size 100 is a round middle ground: 10 (too deep at 1M, 100 000 dirs) and 1000 (too wide per dir at full scale, complicates inspection) are the obvious neighbors. Padding the bucket index to 4 digits keeps directory names sorted lexicographically and matches the worst case at 1M pieces (`0000` to `9999`). The manifest stores the resolved path so the convention can change without a frontend release.
Revisit when: never expected at this granularity. If the puzzle grows past 1M pieces, widen the bucket index padding rather than change the divisor.

### 2026-05-23, backend-realtime, manifest fetched from R2 at boot

Choice: the server reads `MPP_PUZZLE_ID` and `MPP_ASSETS_BASE_URL`, fetches `<base>/<id>/manifest.json` with the global `fetch()` once at boot, and aborts the process on any network error, non-2xx response, or `puzzleId` mismatch. There is no local fallback, no retry loop, and no manifest cache on the host filesystem. The previous `MPP_MANIFEST` path-based env is removed.
Why: a single source of truth for the alpha asset set is the R2 bucket fronted by `assets.millionpiecepuzzle.com`. Removing the on-disk copy in the runtime image removes a class of skew bug (image baked with one manifest, R2 holding another) and reuses the bucket already needed for the per-piece tiles, so the server, the frontend, and the Pages build all read the same files. Fail-fast at boot turns a missing or wrong manifest into an immediate restart loop visible to Coolify, instead of a half-initialized process serving a stale puzzle. The host is allowed to depend on R2 at boot because the R2 custom domain (`assets.*`) is fronted by the Cloudflare CDN and serves an immutable artifact: a cold cache miss is one origin GET, after which the edge serves further restarts. Once Redis is initialized for a given puzzle id the running process no longer needs R2.
Revisit when: the boot-time R2 read becomes a deploy hazard (extended Cloudflare or R2 incident overlapping a deploy). The cheapest mitigation is to keep the last successful manifest on a small persistent volume and read it as a fallback; a heavier one is to publish manifests immutably under a version suffix and pin the env to that version.

### 2026-05-28, qa-and-load, harness PASS criterion bounded to saturation signals

Choice: the load harness flags PASS when (1) no WebSocket close with code 1013 (backpressure) occurred, (2) no `ws` library errors were raised on any bot, and (3) server `error` frames stay below 5% of grab attempts. Grab round-trip latency is reported (p50/p95/p99/max) but not gated.
Why: the Phase 1 exit criterion for this task is "without server saturation", which is the closes/errors/drops dimension. On the alpha CX22 fronted by Cloudflare, the 20-bot/300s baseline against prod recorded p95=3.9s, p99=4.6s, and 24.7% client-side grab timeouts at 5s, while raising zero backpressure closes, zero ws errors, and zero server errors. That multi-second latency was bounded by the writer architecture in place for that baseline, not by a load-induced fault: the dispatch chain processed every message serially (since replaced by [per-group dispatch queues](#2026-05-29-backend-realtime-per-group-dispatch-queues)), the [unthrottled per-pointermove drag broadcast](#2026-05-12-frontend-canvas-drag-no-throttle) fanned out roughly 6 000 outbound writes per second on a sustained 300 drag/s ingest, and the WS writer is a single Node process per the [alpha topology](#2026-05-18-infra-deploy-alpha-topology). Gating PASS on an absolute latency budget would conflate two questions: does the server stay up under load (yes, the harness covers this), and is each interaction snappy (no, and a Phase 2 problem). Keeping the harness focused on saturation lets the verdict track the deployment health it was built to verify and leaves the latency story to its own follow-up.
Revisit when: the remaining writer-perf work lands (write sharding, or the WS writer split off a single process) on top of the pulled-forward per-group queues and drag throttling. Re-run the harness with a latency budget appropriate to the new architecture and tighten the PASS criterion accordingly.

### 2026-06-07, qa-and-load, harness seeds sessions by direct Mongo write

Choice: the load harness seeds one disposable user + database session per bot directly in Mongo (in the Auth.js adapter's document shape) and sends the matching session cookie on each WS upgrade, rather than minting sessions through a dev-only server endpoint. Seeded docs are tagged `loadTest: true` with a per-run `runId`, use reserved `@loadtest.invalid` emails, and are deleted on exit (`--keep-sessions` skips teardown). The cookie name follows the target scheme (`wss` -> `__Secure-` prefix), matching the server's https-only Secure cookie.
Why: the WS upgrade gates on a session resolved against Mongo via the adapter, and sign-in is Google OAuth only with no programmatic path, so a bot cannot authenticate on its own. Direct seeding adds no new server attack surface: a dev-gated session-minting endpoint would be live in prod, where `MPP_DEV_ENABLED=1`, so it would be a standing way to mint authenticated sessions. The cost is coupling to the adapter's collection/field layout (`sessions.sessionToken` + `userId` ObjectId, `users._id`) and needing Mongo reachable from the harness (a tunnel in prod, since Mongo is not publicly exposed). No auth secret is needed: database sessions are an opaque token looked up in Mongo, so the gate accepts a seeded session even when `AUTH_SECRET` is unset.
Revisit when: the Auth.js adapter or session strategy changes (the seeded shape would drift), or a prod path that needs no Mongo tunnel is wanted (then a hardened, separately-secret-gated mint endpoint may be worth its surface).

### 2026-05-28, backend-realtime, scatter as a detached center-dense rounded-square band

Choice: the initial scatter samples each piece body (origin + canonicalOffset) in a rounded-square band detached from the frame, then sets the group origin to body - canonicalOffset. The band lies between two superellipses sharing the frame aspect, exponent `SCATTER_SHAPE_EXPONENT` (4, a rounded square): an inner gap scaled by `SCATTER_GAP_SCALE` (1.4) and an outer halo scaled by `SCATTER_HALO_SCALE` (2.8), both relative to the clear rectangle (frame grown by half a piece) and both >= 2^(1/4) so they enclose it. Each placement draws an angle and a triangular radius across the band (mean of two uniforms, three RNG draws), peaking mid-band and fading to both edges.
Why: pieces render at `origin + canonicalOffset`, and `canonicalOffset` is the solved cell (`col*pieceSize, row*pieceSize`). Randomizing the origin alone left the solved image in place plus bounded jitter (sky pieces high, ground low); randomizing the body instead decouples the shuffled layout from the solved one. The gap leaves empty space around the assembly area so the cloud reads as detached from the canvas, and the triangular radius makes it dense in the middle of the band and dispersing on both sides, like a tipped-out packet of pieces, rather than a uniform band hugging the frame.
Revisit when: the gap around the frame is too small or too large (`SCATTER_GAP_SCALE`), the cloud is too dense or too sparse (`SCATTER_HALO_SCALE`), or the corners read too round or too square (`SCATTER_SHAPE_EXPONENT`).

The sidebar viewer is also kept a sibling of its click target rather than nested inside the `<button>`, since nesting it leaves the viewer blank.

Alongside it, OpenSeadragon's WebGL drawer fails its tile texture uploads ("Error creating texture in WebGL"), leaving the viewer blank. The context2d drawer renders one Deep Zoom image with no measurable cost, so the reference views are reliable regardless of the page's WebGL pressure.

### 2026-06-01, auth-and-accounts, Auth.js via @auth/express with database sessions

Choice: authentication is Auth.js mounted as `@auth/express` on the server's Express app (`app.use("/auth/*", ExpressAuth(authConfig))`), Google the only provider, with database sessions via `@auth/mongodb-adapter` sharing the one `MongoClient`. The session cookie is opaque and resolved against the `sessions` collection; it is set with `Domain` from `AUTH_COOKIE_DOMAIN` (`.millionpiecepuzzle.com` in prod, host-only in dev), `SameSite=Lax`, `httpOnly`, and `Secure` derived from the auth host scheme (`__Secure-` cookie-name prefix when secure). The SPA on `app.*` drives sign-in with the documented form-POST pattern (`GET /auth/csrf`, then a top-level form POST to `/auth/signin/google` with `callbackUrl`), reads `GET /auth/session` over credentialed CORS scoped to `MPP_APP_ORIGIN`, and sets its pseudo through `POST /profile/pseudo`. A `redirect` callback whitelists the SPA origin so the post-login redirect lands on `app.*` rather than the auth host; a `session` callback exposes `user.id` and `user.pseudo`. Indexes the adapter does not create (`users.pseudo` partial-unique, `users.email` unique, `accounts (provider, providerAccountId)` unique, `sessions.sessionToken` unique) are ensured at boot.
Why: Express is the supported host for `@auth/express`, and `ws` + the spectator stream (`/keyframe`, `/events/<t0>`) attach to the same `http.Server`, so one process serves auth, the spectator stream, and the live socket. Database sessions (over JWT) give server-side revocation and a user profile to hang the pseudo on, which attribution needs. One shared Mongo client keeps the adapter and the merge log on a single connection.
Revisit when: a second writer instance is introduced, or sign-in needs more than Google.

### 2026-06-01, auth-and-accounts, WS upgrade requires a valid session

Choice: the WebSocket upgrade is authenticated. `verifyClient` checks the Origin, then resolves the parent-domain session cookie against the adapter's `getSessionAndUser`; an upgrade with no valid session is rejected (HTTP 401 at the handshake). The connection then carries the real `user._id` as `userId` and the profile pseudo, replacing the per-connection `randomUUID()`. Resolving in `verifyClient` (not in the `connection` event) means the connection only fires for an authenticated upgrade, so an early `hello` is never dropped during the async lookup. Spectators are unaffected: they poll `GET /snapshot` and never open a socket. The `setPseudo` WS message and the client-side `localStorage` pseudo are removed; identity is authoritative server-side.
Why: contribution must be gated server-side, not just in the UI. Reading the cookie on the upgrade reuses the same session the SPA already holds (the browser attaches it automatically on the cross-subdomain upgrade), so no token field is added to the protocol. This is the server-side contribution gate the alpha passcode flagged as its replacement: the passcode now only gates the `/play` route during the closed alpha and carries no server weight.
Revisit when: the WS writer is sharded (the per-connection lookup is fine single-process), or the alpha passcode route gate is removed at public launch, or live pseudo change without a reconnect is wanted.

### 2026-06-01, infra-deploy, auth secrets in Coolify env, not in the image

Choice: the auth secrets (`AUTH_SECRET`, `AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET`) are set in the Coolify service environment, never in the Dockerfile or the repo. All non-secret server config (`AUTH_URL`, `AUTH_TRUST_HOST`, `AUTH_COOKIE_DOMAIN`, `MPP_APP_ORIGIN`, `MPP_ALLOWED_ORIGINS`, puzzle id, asset base, rate-limit tunables) is baked as Dockerfile `ENV` with production values. `docker-compose.yml`, the file Coolify deploys, carries only the infra wiring (`MPP_PORT`, Redis/Mongo URLs) and the `${VAR}` passthrough for the three secrets; it does not re-declare the non-secret keys. `docker-compose.override.yml`, which `docker compose` auto-merges for local runs but Coolify does not deploy, overrides those baked values with localhost equivalents for local dev.
Why: a deliberate exception to deployment-config-in-code for the secrets. Baking OAuth credentials or the session secret into the image (built in CI, possibly cached or shared) leaks them; they belong in the orchestrator's secret store. The non-secret host/cookie config has no such risk and stays in the image so a change is one commit plus a rebuild. The split into the override file is load-bearing: a compose `environment:` entry overrides the image `ENV`, so re-declaring the non-secret keys in `docker-compose.yml` with localhost defaults silently shadowed the baked prod values in production (broken CORS and a host-only session cookie) until the values were duplicated into the Coolify env. Keeping those keys out of the deployed compose file removes both the shadowing and the duplication: prod needs only the three secrets in Coolify.
Revisit when: a real secrets manager is introduced (e.g. for rotation), at which point all secrets move there together; or Coolify is reconfigured to merge the override file, which would reintroduce the shadowing.

### 2026-06-01, auth-and-accounts, login anti-abuse is internal per-IP rate limiting

Choice: login abuse is bounded by two Redis-backed per-IP fixed-window counters (`INCR` plus a first-hit `EXPIRE`): a generous window on all `/auth` and `/profile` requests, and a stricter window on the OAuth callback (`/auth/callback/google`, the GET redirect Google sends back), the account-creation chokepoint. Over budget returns 429. The limiter fails open on a Redis error so an outage cannot lock everyone out of signing in. No Cloudflare Turnstile.
Why: with Google-only OAuth, a `users` row is written only on a successful Google callback, so each fake account costs the attacker a real Google account, an abuse cost Google already enforces. Turnstile's classic value (defeating password brute force) does not apply with no password form. The cheap remaining surface is request-rate abuse on the auth endpoints, which per-IP limiting addresses directly and reuses infrastructure already present (`clientIp`, fixed windows). Redis-backed, not the in-memory `TokenBucket` the high-frequency WS path keeps, because auth traffic is low-volume and a counter in Redis stays correct if the writer is later sharded.
Revisit when: public traffic arrives. A shared NAT counts against one IP budget and an attacker rotating IPs evades it; the documented next escalation is a Cloudflare edge Rate Limiting Rule on `/auth/*` (out of scope here). Tune the windows against observed legitimate login rates.

### 2026-06-03, frontend-canvas, viewport-driven texture streaming

Choice: per-piece AVIF textures and their Pixi nodes are built on demand instead of all at `build()`. `build()` now creates only lightweight group containers and the spatial index for every group, with each group's `localBounds`, minimap dots and membership derived from geometry (canonical offsets), so a group is fully described while dehydrated (no textures fetched). A streaming engine then hydrates (fetches textures, builds nodes) groups inside a hydrate ring around the viewport and frees them (destroys nodes, `Assets.unload`) only when they leave a wider keep ring (hysteresis, `HYDRATE_MARGIN_FRAC` 0.3 vs `DEHYDRATE_MARGIN_FRAC` 0.9). Residency is a pure function of the viewport window, not of LOD tile coverage: a window piece stays resident even while hidden behind a baked tile. Loads drain a bounded `HYDRATE_MAX_INFLIGHT` (128) per frame from a queue, and `bakeTile` defers (re-enqueuing the missing groups) until a cell's non-held clusters are hydrated, so a tile never bakes blank. `build()` resolves once every group in the hydrate ring is hydrated (the first viewport's pieces are on screen; the LOD bakes its tiles from them within the next frames), driven by the existing per-frame ticker; progress is reported in coverage units (groups), not the 1M piece set. `applySnap` and `applySnapshot` move membership (`pieceIds`, `pieceToGroup`) unconditionally and reparent built nodes only when present, so an off-screen merge updates the model with no textures resident.
Why: a `Promise.all` over every per-piece AVIF and one node per piece at `build()` is O(board) in fetches, VRAM and main-thread work, which does not reach 1M. Bounding both to the visible window is the scaling move. Residency is window-only, not tied to LOD tile coverage, because freeing a piece the instant its tile bakes would thrash against a spectator keyframe re-base that un-baked every tile. The re-base therefore dirties only the tiles of the clusters it actually changes (membership, position, locked), a targeted diff over the model it already maintains, not every tile (see the spectator keyframe entry): an unchanged re-base re-bakes and re-fetches nothing, and the build coverage (gated on hydration, not tiles) stays monotonic. Keeping window pieces resident makes a re-bake draw from still-resident pieces with no blank and no re-fetch. "No per-piece fetch when zoomed out" is therefore read as "no eager whole-board fetch": a zoomed-out view fetches only its bounded window (~15-25k pieces at 1M, far smaller than the board) and keeps it resident, chosen with the product owner ("load the visible pieces on demand, just more of them at max zoom-out, always fewer than the whole board") over a source-pyramid scattered-remap bake (a larger image-pipeline-coupled feature) and over flat-color placeholder tiles (which degrade the overview at event start). Geometry-derived group state keeps the spatial index, minimap and merge math correct without any texture resident.
Revisit when: resident VRAM at a deep zoom-out is now bounded: the covered clusters whose tiles have gone idle are freed instead of held (see [hot-tile residency](#2026-06-05-frontend-canvas-hot-tile-residency-frees-covered-cold-clusters)), so the window no longer stays fully resident (~15-25k full-res textures was the cost traded away here). The group-container and index population now runs in yielding chunks behind a determinate `build` status phase, and per-piece geometry is lazy (only hydrated pieces generate edges); see [chunked board build](#2026-06-04-frontend-canvas-chunked-board-build-with-lazy-geometry). Known cost: a background tab parks the ticker, so the initial fill (and thus the loading cover) pauses until the tab is focused.

### 2026-06-04, frontend-canvas, chunked board build with lazy geometry

Choice: the post-download `build()` runs its construction in time-budgeted yielding passes instead of one synchronous block, behind a `build` loading phase, and per-piece edge geometry is lazy. `chunkedPass` runs a step in bursts capped at `BUILD_CHUNK_BUDGET_MS` (8 ms) with a `setTimeout(0)` macrotask yield between bursts, reports cumulative progress, and bails when a `buildToken` (bumped at `build()` start and in `clearWorld()`/`destroy()`) changes, so a teardown or rebuild mid-construction stops the in-flight passes. Three passes: A piece->file map, B piece->group membership, C one dehydrated container per group plus its spatial-index entry. The world is kept `renderable = false` across A-C so Pixi does not traverse the partially built subtree each frame between yields, restored before `fitView`. Full edge geometry (`generatePieceGeometry`, extracted from `generatePuzzle` in `@mpp/shared` with identical output) is generated on demand only for hydrated pieces and cached (`geomFor`), while index bounds and the minimap use `canonicalOffsetFor` (offset derived arithmetically from the id), so neither generates edges. Progress is reported as `{ phase: "build" | "textures", loaded, total }`: build counts pieces + groups across A-C, textures wraps the existing first-viewport coverage.
Why: at 1M the eager path froze the main thread before the (already chunked) texture stream resolved: `generatePuzzle` over every piece (~760 ms measured at 1M), the map builds (~300 ms), and the group/index loop (multiple seconds), all synchronous, with the loading cover stuck on the texture phase at 0% until it unblocked. Full edge geometry is only needed at hydration (~the visible window, ~4 000 pieces), so generating all 1M up front was both a freeze and wasted work; lazy generation removes it and matches the viewport-streaming design. Chunking the remaining construction keeps the main thread responsive and lets the cover paint determinate build progress. Determinism is preserved by construction: `generatePieceGeometry` is the same code factored out (each shared edge keys its own subseed by grid position, so it is order-independent), verified byte-identical to the eager generator's output.
Revisit when: resident VRAM at a deep zoom-out window still folds into "Rendering stays smooth on commodity hardware at 1M". The one-time full-world sort and the per-frame sort are gone (the world no longer uses `sortableChildren`, see [z-order layers](#2026-06-04-frontend-canvas-z-order-layers-replace-world-sortablechildren)), and the geometry cache is now evicted on dehydrate (same entry).

### 2026-06-04, frontend-canvas, z-order layers replace world sortableChildren

Choice: `world` no longer uses `sortableChildren`. A group's stacking depth is now which fixed layer container holds it. `world`'s children are added once in stacking order: the backdrop fill, the frame, `lockedLayer`, `unlockedLayer`, the LOD tile container, `remoteHeldLayer`, `localHeldLayer`. A group is created into `lockedLayer` or `unlockedLayer`; a local grab reparents its container into `localHeldLayer` and a remote grab into `remoteHeldLayer`, and a drop, release or anchor reparents it back to its resting layer. `placeGroupInLayer` skips the move when the group already sits in the target layer. Separately, the per-piece edge geometry cache (`geomById`) is evicted in `destroyPieceNodes`, so it holds at most the resident pieces rather than every piece ever hydrated.
Why: with `sortableChildren`, every change to a group's `zIndex` (each grab, drop and anchor) marked `world` sort-dirty, so the next render re-sorted all its children, an O(N log N) pass over the whole board, and the first render after `build()` paid that full sort once over 1M containers. Fixed layers express the same stacking by child order, so a depth change is a reparent (guarded to O(1) when unchanged, otherwise an O(old layer size) array splice paid once per grab/anchor, not per render) and no render ever sorts. The geometry cache grew unbounded because `geomFor` only ever inserted; since it regenerates a piece's geometry deterministically on re-hydration, evicting on dehydrate bounds it to the visible window like the textures and nodes, which matters over a long deep-zoom session.
Revisit when: a grab at 1M ever hitches on the array splice that reparents a group out of a million-child layer. The splice is one occasional frame cost that replaced a per-render full-board sort, so it is the better trade; if it still shows, back the layers with a structure that removes in O(1).

### 2026-06-05, frontend-canvas, hot-tile residency frees covered cold clusters

Choice: when the zoom-out LOD covers the board, a non-held cluster whose tiles are all baked and "cold" has its per-piece textures and Pixi nodes freed; the baked tiles keep drawing it. A tile cell is hot for `LOD_HOT_TTL_MS` (9 s) after its last change, recorded in a `cellDirtyMs` map updated at the single dirtying chokepoint (`markTilesDirty`, fed by every drop, snap, rollback, grab, the spectator keyframe re-base diff, and each spectator drop-glide frame). `isCoveredCold(group)` is: the initial fill resolved, LOD active, not held, every occupied tile baked (`allCellsReady`), and no occupied cell hot. The initial-fill guard matters because the loading-cover gate counts hydrated groups and a cold start has no hot cells, so freeing covered clusters mid-fill would stall the cover; once the first viewport is up, the idle window frees. Freeing runs in two places: `reconcileGroupResidency` (so `updateResidency`, on every camera change, and a post-merge `applySnap`, neither hydrate nor retain a covered-cold cluster) and a throttled per-frame `sweepColdResidents` (one pass per `LOD_COLD_SWEEP_FRAMES`, 6) that frees residents that cooled in place and prunes stale `cellDirtyMs` entries. Held clusters stay resident; with the LOD inactive (zoomed in) nothing is covered, so the whole window hydrates and renders live as before. Freeing a cluster never dirties its tiles, so a cold region settles to tiles-ready, clusters-freed, zero work.
Why: the viewport window at a deep zoom-out is ~15-25k pieces at 1M, and holding all of them resident (full-res AVIF plus nodes) behind the baked tiles was the unbounded-VRAM cost the [viewport-driven texture streaming](#2026-06-03-frontend-canvas-viewport-driven-texture-streaming) entry traded away. Per-cell hotness, not per-group, is the right unit because `bakeTile` renders a cell from all its non-held clusters: any cluster sharing a cell with a recent change must stay hydrated for that cell to re-bake without a blank, and marking the cell hot keeps exactly those clusters resident. The targeted re-base diff is the prerequisite (a whole-set dirty on every keyframe re-base would keep every cell hot forever); with it, an idle region's cells cool once events stop touching them and free. Resident VRAM at a deep zoom-out then converges on the hot working set plus the bake-in-progress and held clusters, not the window.
Revisit when: the first change to a long-cold region re-hydrates its tile and can flash one blank tile (about one fetch) before settling; if that shows on the alpha, serve the stale baked tile until the re-bake lands (keep the sprite visible in `markTileDirty` until the next `markBaked`), trading the blank for one or two frames of stale positions, imperceptible at 12-24 px per piece. Tune `LOD_HOT_TTL_MS` against the observed event cadence and working-set size. Capping pieces per LOD tile (ROADMAP backlog "Cap maximum piece on tiles") still bounds the per-tile bake cost, which this does not address.

### 2026-06-05, backend-realtime, keyframe reads off the write queue

Choice: the spectator keyframe (`KeyframePublisher.regenerate` -> `buildKeyframe`) reads the board directly off `RedisState`, outside the per-group dispatch queue, and its `readAllPieces` / `readAllGroups` are independent ioredis pipelines (not a MULTI). A keyframe can therefore capture a merge mid-apply (`applyMerge` writes pieces and the surviving group in several non-atomic ops): a piece reassigned to its new group before that group is repositioned, or pointing to a group already deleted. Under protocol v3 the contributor no longer reads the whole board on connect (welcome carries no board); its per-viewport `region_state` construction runs inline off the dispatch queue, like this keyframe read, so it too is best-effort and can observe a group mid-merge.
Why: the read/write split exists so spectator reads never load the write path. Routing the keyframe through `runGlobal` would stall every gameplay write behind a full-board read, and a truly atomic read (MULTI/Lua over the whole board) would block single-threaded Redis just as hard; both fight the architecture at 1M pieces. The artifact is cosmetic and self-healing on two fronts now: the client tolerates a piece whose group is missing (`puzzleStage` builds it with `idsByGroup.get(id) ?? []` and skips a reparent when `!host || !from`), the very next event window replays the merge in `seq` order (the snap reaches the client within the interpolation delay), and the next keyframe re-base corrects any residue. A `region_state` that catches a merge mid-apply heals the same way: the global snap the contributor also receives and the next `region_state` for the cell reconcile it.
Revisit when: spectator-visible partial-merge artifacts become objectionable despite the window replay healing them within the interpolation delay. Reading the keyframe under a short-lived snapshot (Redis `MULTI`/`WAIT` or a read replica) is the next step before a global barrier.

### 2026-06-06, backend-realtime, viewport-scoped initial state on join

Choice: a contributor's `welcome` carries no board (protocol v3). The client builds an empty board and streams groups in per viewport via `region_state`, so the join payload is bounded by the viewport, not the piece count. `welcome` is followed only by `activity`, `leaderboard`, and one `minimap`; `handleHello` no longer reads the board and runs inline (no global barrier). The first bounded `viewport` (and every later pan entering new broadcast cells) triggers a `region_state` construction stream from the [group index](#2026-06-06-backend-realtime-group-spatial-index-and-pan-resync): each entry is the group ORIGIN, locked, size, and piece ids (the singleton's own id, or a Redis read for a size>1 group). A default fit viewport is a global subscription and streams nothing by design (sending it the whole board is exactly the full-state push being removed); the minimap carries the overview until the user zooms in past `MPP_BROADCAST_MAX_CELLS` (~256 cells, roughly a 256x256-piece window). `build()`'s whole-board construction is kept (the spectator builds its full keyframe board through it); its per-group Pass C is extracted into a shared `constructGroup` reused by `build()` and `applyRegionState`. The client stage is made partial-board-safe: `applySnap` moves membership for every added piece unconditionally (so a known host gets complete membership even when a source group was never visited) and, when the host itself is unknown, reassigns the pieces and removes only the known source groups (no phantom, the merged cluster is built wholesale by the next `region_state`); `applyRegionState` builds an unknown group and additively reconciles a known group's membership/locked. On a server-driven rebuild (welcome resent on reset/force-complete) the hub resets the client's cell subscription so its next viewport re-streams the region (the connection persists but the board was discarded).
Why: the full `state` array on connect was O(board) (~25 MB at 1M), the last unscaled state-delivery path after the spectator stream and the pan resync. Bounding it to the viewport reuses the read model and the construction path the pan resync already built. It is accepted that the client's spatial index, LOD coverage and group containers build up per visited region during play, and that the honest landing experience at scale is an empty board plus the minimap, filling in on zoom/pan, since active solvers are zoomed in to place pieces while passive far-zoomed viewers read the spectator HTTP stream. Keeping `build()`'s whole-board pass (not deleting it for the contributor) is what lets the spectator keep its keyframe board through the same `constructGroup`, so one construction path serves both transports. The partial-board `applySnap`/`applyRegionState` fixes are required because a remote merge can straddle the visited boundary, which full-state-on-join never exposed.
Revisit when: the per-cell construction payload (full cells, not just changed groups) bites; or a zoomed-out contributor wants the source image as a backdrop (ROADMAP backlog: render the spectator keyframe as the zoomed-out contributor backdrop).

### 2026-06-06, backend-realtime, server-computed minimap grid

Choice: the minimap is rendered from a compact server-computed density grid (`buildMinimapGrid` in `@mpp/shared`, pure), not from the client's (now partial) local board. The grid bins every piece by its body center into a play-zone-aligned grid (~96 cells on the larger axis), splitting loose from locked counts. One builder, two delivery channels: it is added to the spectator keyframe (spectators already download it), and broadcast to contributors as a small periodic `minimap` WS message tied to the keyframe regenerate (so no second full-board read) plus one on join. The client draws the grid as the global overview and overlays its locally known groups (the visited regions it has fresh positions for) on top, refining the coarse grid; the overlay only ever adds, never zeroing a coarse cell it knows only partially.
Why: the contributor minimap could no longer iterate the whole board (it only holds visited regions under the viewport-scoped join), so the global overview must come from the server. Downloading the full keyframe just for a minimap would defeat the bandwidth goal, hence the small grid over WS instead. Tying the broadcast to the keyframe cadence (default 5 min, with the idle gate) keeps it cheap; the live local-known-groups overlay covers own-region freshness in between, so the coarse cadence is only felt in regions the contributor is not looking at.
Revisit when: the 5-minute contributor minimap cadence is too stale for distant-region progress (push the grid on a lighter, more frequent channel), or the ~96-cell resolution reads too coarse at 1M.

### 2026-06-08, frontend-shell, interested counter dedup by hashed-IP set

Choice: the landing "I'm interested" count is a Redis SET per puzzle (`puzzle:{id}:interested`), one HMAC-SHA256-hashed visitor IP per member. `POST /interested` SADDs the hash and returns SCARD; `GET /landing` returns SCARD plus a SISMEMBER for "me". The count is the set cardinality, so dedup is inherent with no separate counter, and there is no request body or query string (the IP is derived server-side via `clientIp`), keeping both endpoints CORS simple requests behind the spectator per-IP guard. The HMAC is keyed by `AUTH_SECRET` (the existing per-deployment secret, read straight from `process.env`, not copied into `config`); a missing key in dev falls back to a fixed salt so local dedup stays stable across restarts. Both handlers fail open on a Redis error (zeroed count, optimistic `me`), like the spectator guard and rate limiters.
Why: a SET makes "unique opt-ins" a property of the data, not a counter that can drift, and SADD/SCARD/SISMEMBER are each O(1). The IP is hashed (not stored raw) because, unlike the per-IP rate-limit keys that expire within seconds, this set is permanent; keying the HMAC by the deployment secret makes the digest irreversible even though the IPv4 space is small, so a Redis dump exposes no addresses. Reusing `AUTH_SECRET` avoids introducing a second secret to manage; the context-tagged HMAC message (`interested:<ip>`) domain-separates it from any other use.
Revisit when: the set cardinality grows large enough that its memory matters (millions of distinct opt-in IPs), or the writer is sharded such that a per-IP guarantee needs more than this Redis-backed set (it is already shard-safe). Bucketing IPv6 by `/64` is the same open follow-up the spectator rate limit carries.

### 2026-06-08, infra-deploy, scheduled backups to a private R2 bucket

Choice: a `backup` sidecar service (`docker/backup/`, Alpine + `mongodb-tools` + `redis` + `rclone`) runs a loop that every `MPP_BACKUP_INTERVAL_SEC` (21600, 6h) writes a gzipped `mongodump --archive` of the `mpp` database and a Redis RDB pulled over the wire (`redis-cli --rdb`, so the data volume is not shared into the container), pushes both under dated keys (`mongo-<ts>.archive.gz`, `redis-<ts>.rdb.gz`) to a private R2 bucket `mpp-backups`, then prunes each kind to the newest `MPP_BACKUP_KEEP` (3). The rclone `r2` remote is configured entirely from env (`RCLONE_CONFIG_R2_*`): type/provider/acl baked in `docker-compose.yml`, the endpoint (which carries the R2 account id) and the access key id/secret passed through from the Coolify service env, never committed. `docker-compose.override.yml` sets `MPP_BACKUP_ENABLED=0` so the loop idles locally (no R2 creds there). The loop runs the first pass at container start, logs ok/FAILED per pass, and a failure never exits the container. Restore steps live in the `backup.sh` header.
Why: the single CX22 keeps all live state in the Redis and Mongo Docker volumes, so a host or volume loss with no off-host copy is unrecoverable. A sidecar in the deployed compose file keeps the backup config in code and ships with the stack, where a host cron would not be captured by the repo. The bucket must be private and separate from `mpp-assets`: that bucket is public-read over `assets.millionpiecepuzzle.com`, and a Mongo dump carries user profiles (pseudo, OAuth email), so it cannot share the public asset lifecycle. Keep-3 is deliberately shallow: the dumps are tiny (Redis ~16-32 MB, the merge log modest) so storage is not the constraint, a 3-deep window covers the dominant failure (host/volume loss, where any recent backup restores), and a 6h cadence bounds data loss to one interval. The Mongo merge log is the durable record of solved structure; a Redis-only loss (loose non-merging-drop positions) is degraded but rebuildable from it, so Mongo is the higher-value dump.
Revisit when: the writer/state is sharded (one sidecar per data store, or a backup that understands the shard map), point-in-time recovery is needed (the keep-3 6h snapshots are coarse recovery points, not a continuous log), or a restore drill shows the RDB/archive round-trip needs more than the documented manual steps. If R2 storage or egress ever matters, the dumps are small enough that raising keep-N stays cheap.

### 2026-06-09, piece-generation, edge uniqueness validated per-seed

Choice: "every shared edge unique" is validated per base seed by counting distinct 32-bit edge subseeds across the ~2M shared edges of the 1000x1000 board (a streaming `validateGeneration`, run at full scale via `npm run validate:generation`), not guaranteed by entropy. The edge seed space was left at 32 bits because `subseed` is empirically injective on the grid lattice (0 collisions across sampled seeds, 1 998 000 distinct seeds for 1 998 000 edges). Uniqueness is measured on the seed, not the param tuple: each seed drives a deterministic mulberry32 stream, so distinct seeds yield distinct silhouettes and so distinct pieces, and a u32 set stays cheap at 2M where a float-tuple set would not.
Why: a naive birthday estimate over 2M values in 2^32 predicts ~465 collisions, but `subseed`'s Knuth-multiplicative mixing acts as a near-permutation on the small structured (domain, row, col) lattice, so in practice there are none. Counting seeds is exact and cheap; widening the seed (e.g. combining two subseeds into 53 bits) would re-roll all geometry for a problem that does not occur.
Revisit when: the production seed reports any duplicate (run `npm run validate:generation --seed <real>` before slicing, since the source AVIF tiles are pre-masked from this geometry and a seed change forces a re-slice), or the grid grows enough past 1000x1000 that 32-bit collisions become likely; widen the edge seed to 53 bits then, before the gigapixel slice.

### 2026-06-11, backend-realtime, per-tile piece cap on non-merging drops

Choice: a non-merging drop is rejected when its destination cell already holds more than `MPP_TILE_PIECE_CAP_MULTIPLIER` (default 8) times the cell's solved density (the pieces that fill one cell when solved, `(WORLD_TILE_SIZE / pieceSize)` squared), or an absolute `MPP_TILE_PIECE_CAP` when that is set (a testing/ops escape hatch). The cell is the shared world grid cell `WORLD_TILE_SIZE`, which is exactly one LOD tile and one broadcast cell, so the capped region is precisely the tile the client sees: the red rejection flash and the loading badge are drawn on the same grid, with no mismatch between what the overlay shows and what the cap rejects. `handleDrop` sums the member counts of the groups in the destination cell from the `GroupIndex` (excluding the dropped group), and if adding the cluster would exceed the cap, releases the hold and emits a `rollback` to the pre-drag position: `reason: "tile_full"` to the dropper (which flashes that tile red and toasts "Too many pieces on this tile") and a plain position correction to the neighbours who watched the drag (excluding the dropper). Merges and frame anchors are exempt: a merge removes a loose cluster and an anchor locks to the frame, so neither grows the loose pile a zoom-out tile must bake.
Why: the zoom-out LOD bakes a cell's live pieces into one texture, so an unbounded pile on a single tile (e.g. all 1M pieces) would make that bake O(board) and defeat the LOD's constant-per-piece cost. The server already keys every group's position in the `GroupIndex`cell with its member count, so the cap is a cheap per-cell sum over the existing read model with no new state. Expressing it as a multiple of the solved density lets the cap scale with the cell size instead of being a magic absolute. The reused, previously-unemitted`rollback`message carries the bounce; the reason field separates the cap rejection (visible feedback) from a plain correction.
Revisit when: 8x proves too loose or strict in playtests (tune `MPP_TILE_PIECE_CAP_MULTIPLIER`); also note a cell pushed over the cap by exempt merges blocks even an in-cell rearrange of a loose cluster.
