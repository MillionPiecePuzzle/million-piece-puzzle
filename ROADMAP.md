# Roadmap

Three phases, eleven tracks. A phase is closed only when its exit criterion is met. Tasks are not estimated. Each task carries an exit criterion, not a description.

Statuses: `[ ]` not started, `[~]` in progress, `[x]` done.

## Tracks

1. `shared-protocol`: shared TS types, WS messages, piece and user schemas
2. `piece-generation`: Bezier silhouettes, snap geometry, piece types
3. `image-pipeline`: libvips processing, Deep Zoom tiles, per-piece AVIF, R2 upload
4. `frontend-shell`: landing, routing, spectator/contributor modes, auth modal
5. `frontend-canvas`: PixiJS rendering, OpenSeadragon, drag/drop, LOD, culling
6. `backend-realtime`: WS server, drag/drop/snap logic, Redis state, Mongo logs, CDN snapshots
7. `auth-and-accounts`: Auth.js (Google), pseudo onboarding, sessions, login anti-abuse
8. `infra-deploy`: Docker, Coolify, Hetzner, Cloudflare (Pages, R2, CDN, DNS)
9. `tooling-foundations`: monorepo workspaces, shared tsconfig, eslint, prettier
10. `qa-and-load`: load tests up to 1M simulated pieces and clients
11. `legal`: privacy policy, ToS, GDPR notes, license attributions

---

## Phase 0, Local MVP, CLOSED

**Exit criterion (met)**: a single user completes a puzzle of N configurable pieces in a browser, with the full architecture running locally (WS server + Redis + Mongo via docker-compose), in anonymous mode.

Delivered across all tracks: monorepo with shared tsconfig, lint, format, build end to end; WS message types and piece/anonymous-user schemas in `shared`; procedural Bezier generator with matching tabs/blanks and snap geometry; local AVIF slicer plus manifest; authoritative WS server applying drag/drop/snap with Redis state, Mongo snap log, docker-compose; landing and play routes with a spectator-to-contributor entry; PixiJS canvas with drag/drop/snap feedback, snap and end-of-puzzle animations, completion detection; anonymous ephemeral-id sessions. Polish: frame-based anchoring, confetti loop and congrats modal, silhouette tuning, `/play` chrome (topbar, zoom rail, Leaderboard and Activity panels, adaptive grid). Two code audits cleared: repo green (lint/format/typecheck), doc drift fixed, dispatch queue, disconnect cleanup on-queue, `detectSnap` alignment fix, WS error surfacing, deterministic-core unit tests, pipelined initial Redis write.

## Phase 1, Closed Alpha, CLOSED

**Exit criterion (met)**: 5 to 20 invited people, connected concurrently, complete a 10 000-piece puzzle on a deployed instance, in anonymous mode.

Delivered across all tracks: protocol version field in the handshake plus presence messages (join, leave, viewport, cursor); generation validated and stable at 10 000 pieces; Deep Zoom tile pyramid and per-piece AVIF set on R2 with a manifest; landing with a single enter-canvas CTA and a spectator/contributor split where the spectator renders from polled `GET /snapshot` (no WS budget) and Contribute upgrades to a WebSocket, a `puzzleId` change forcing a clean rebuild; OpenSeadragon reference panel, frustum culling smooth at 10 000 pieces, mini-map global progress, collaborator cursors from presence; viewport-neighbor broadcast scoping for drag and drop (snap stays global), periodic CDN snapshots, dispatch-boundary message validation, and a hardened WS boundary (Origin allowlist, payload cap, per-connection token bucket, backpressure close); anonymous pseudo entry attached to the session; server on Hetzner via Coolify, Cloudflare front (Pages `app.`, backend `ws.` with DNS and SSL at Coolify), CDN edge cache for snapshots, R2 buckets for tiles and textures; unit tests for the merge-and-anchor path and the per-group dispatch queue plus a 20-client load test on a 10 000-piece puzzle without saturation.

Complementary: closed-alpha gate (landing passcode, single `alpha-3` puzzle from R2, completion freezes the board), env-gated dev controls usable from any session mode, floating Contribute CTA, throttled `viewport` presence, live leaderboard and backfilled activity ticker, server-computed play-zone hard limits (camera clamp, darkened out-of-bounds, 15% min zoom, minimap with frame/pieces/frustum), scatter shaping decorrelated from the solved image, reference enlarge modal, staged load with progress, and clean reset/complete handling carried into the spectator snapshot. Bugs fixed: transient errors no longer fatal, cleared display objects destroyed, same play zone on all clients, in-flight `build()` guarded against rapid `state` changes, and the stale board (orphaned LOD sprite) cleared on a mode-switch rebuild.

Performance pulled forward from Phase 2, built as the real solution and kept at 1M scale: drag broadcasts coalesced to one per frame, per-group dispatch queues, a zoom-out LOD, and a per-IP rate limit over the per-connection token bucket. Viewport and write sharding stay deferred, blocked on the single-writer alpha topology rather than on piece count.

## Phase 2, Public 1M

**Exit criterion**: the puzzle is open to the public, with 1 000 000 pieces on a single shared canvas, full auth, monitoring sufficient to operate, and legal documents in place.

### `shared-protocol`
- [ ] Protocol frozen at v1, breaking changes go through version bump
- [x] `eventStartsAt` (unix ms) included in `welcome` so clients can sync the cascade trigger. Sourced from `MPP_EVENT_STARTS_AT` (default 0 = no scheduled start), and mirrored into the spectator `Snapshot` since its synthetic `welcome` derives from it
- [x] Anti-programmatic-solving: randomized wire ids + server-only seed (cross-track: `backend-realtime`, `image-pipeline`, `frontend-canvas`). A connected client can no longer reconstruct adjacency: ids are seed-permuted (`wireId = P(gridId)`, `packages/shared/src/permutation.ts`), a group's transmitted position is its anchor piece's world position with per-piece grid-unit `(dx, dy)` offsets, and the `generationSeed` is gone from the public manifest (server-only via `MPP_GENERATION_SEED`), so neither `id % cols` nor seed-regenerated geometry reveals the solution. The server keeps its grid-id + internal-origin model with translation only at the wire boundary (`packages/server/src/wire.ts`); R2 tiles and the manifest are keyed by wire id; the client renders from pre-masked/pre-bordered tiles plus the offsets, holding no geometry or seed (snap flash degraded to a tile-shaped glow). `PROTOCOL_VERSION` 3 -> 4, `SPECTATOR_FORMAT_VERSION` 1 -> 2. Verified by unit tests (permutation bijection/stability, `wire.ts` id+position round-trip, member offsets, locked-cluster solved position) and a synthetic re-slice (manifest carries no seed, wire-id tile paths resolve). Taking it to prod re-uploads the R2 set under wire-id paths and sets `MPP_GENERATION_SEED` to match the slice `--seed`. See DECISIONS: anti-programmatic-solving

### `piece-generation`
- [x] Generation pipeline produces and validates 1 000 000 unique pieces: a streaming `validateGeneration` (`packages/shared/src/generator/validate.ts`) walks the 1000x1000 grid row by row, holding only the current and previous row (never the full 1M array), checking ids, canonical offsets, border-flat/interior-curved edges, the eight curved-param ranges, neighbour sign+param agreement, and edge uniqueness (distinct seeds across the 1 998 000 shared edges, which by mulberry32 determinism implies distinct silhouettes and so distinct pieces). The 32-bit `subseed` is empirically injective on this lattice (0 collisions), so no widening was needed and the validator guards it per-seed. Run at full scale via `npm run validate:generation` (1000x1000 in ~1.8s: 1 000 000 pieces, 1 998 000 unique edges, no violations); the vitest suite's 10 000 check was re-pointed onto the same validator so `npm test` stays fast. See DECISIONS: edge uniqueness validated per-seed

### `image-pipeline`
- [x] Piece border baked into the tile: the slicer strokes the silhouette path (shared `PIECE_BORDER_*` style in `generator/path.ts`) over the alpha-cut window before AVIF encode and sets `borderBaked: true`; the frontend drops its render-time per-piece stroke when the flag is set (kept for slices not baked with a border), so the outline ships in the tile, costs nothing at render time, and the zoom-out LOD bakes pick it up for free. Trade-off accepted: changing the border color/thickness now needs a re-slice and a full re-upload (every tile changes), and the border is no longer toggleable at runtime. See DECISIONS: pre-cut alpha masks baked server-side
- [x] Gigapixel source processed end to end (Deep Zoom + per-piece AVIF, pre-masked alpha-cut server-side) and hosted on R2. The slicer reads the source as a real file via libvips random access (no global RAM buffer), bakes each piece's bezier silhouette into its AVIF alpha (the piece path from `@mpp/shared` rasterized and `dest-in`-composited over the per-piece window), and `dzsave`s the reference pyramid straight from the file; the manifest carries `premasked: true` and the frontend drops its render-time silhouette mask, showing the alpha-cut tile as-is (geometry still drives layout, the spatial index, the minimap and the snap flash). A procedural synthetic source (`scripts/synthetic-source.ts materialize`, `npm run materialize`) writes a real tiled BigTIFF (flat, not pyramidal: 64-bit offsets for the >4 GB gigapixel raster, slicer builds its own reference pyramid), chunk-rendered to bound RAM, that the slicer treats exactly like a final photo, so the same code path runs for the gigapixel image by swapping the source file. Validated end to end on a materialized synthetic source: alpha-cut AVIF pieces interlock at their solved offsets, tabs clear the tile margin, and the Deep Zoom reference pyramid renders. R2 hosting uses the existing `npm run upload` (now `--checksum`, so an identical re-slice re-pushes nothing); uploading the final gigapixel asset is the deploy step. See DECISIONS: pre-cut alpha masks baked server-side, and synthetic source: banded assembly, deflate, ~6 GP machine ceiling

See DECISIONS: /play entry gate mirrors the scheduled start

### `frontend-canvas`
- [x] Zoom-out LOD scales to 1M: client-baked tile cache with constant per-piece density, cull and per-tile bake bounded by the visible window via a spatial index (`groupGrid.ts`, `lodTiles.ts`). Verified at the alpha's 10 000; the one-time index population at build now runs in yielding chunks (`Chunked board build`), so it no longer freezes the main thread, and full 1M runtime verification is folded into `Rendering stays smooth on commodity hardware at 1M`
- [x] Viewport-driven texture streaming: per-piece AVIF textures and nodes are hydrated on demand for groups within a viewport ring and freed when they leave the wider keep ring, so entering `/play` fetches only the visible window (verified at the alpha's 10 000: a fit fetched and built exactly the ~4 095 in-window groups of 10 000, monotonically, then panning away freed them all and returning re-queued them), and the zoomed-out view bakes its tiles from that resident window. Verified at the alpha's 10 000; bounding the resident VRAM of a deep zoom-out window is handled in `Rendering stays smooth on commodity hardware at 1M` by freeing covered idle clusters, and the one-time O(board) group-container and index population now runs in yielding chunks (`Chunked board build`), with full 1M runtime verification folded into `Rendering stays smooth on commodity hardware at 1M`
- [x] Chunked board build with full status coverage: the post-download `build()` work runs in time-budgeted yielding passes (piece->file map, piece->group membership, one dehydrated container per group + spatial-index entry) behind a `build` status phase, and per-piece edge geometry is generated lazily (only for hydrated pieces) instead of eagerly for the whole board, with index bounds and the minimap using the canonical offset derived arithmetically from the id. The loading cover walks connect -> manifest -> build (determinate) -> textures (determinate) -> ready with no frozen gap, and construction never blocks the main thread. Verified at the alpha's 10 000 (cover walked every phase determinate, board on screen, minimap complete) and determinism preserved (lazy per-piece geometry identical to the eager generator)
- [x] Rendering stays smooth on commodity hardware at 1M pieces. The world dropped `sortableChildren` for fixed z-order layer containers, so neither the first render nor any later frame sorts the whole board; the per-piece edge-geometry cache is evicted on dehydrate (see [z-order layers](DECISIONS.md#2026-06-04-frontend-canvas-z-order-layers-replace-world-sortablechildren)); and resident VRAM at a deep zoom-out is bounded by freeing covered idle clusters: the spectator snapshot dirties only the tiles of clusters it actually changes, and a non-held cluster whose tiles all go cold is freed so only the hot working set, the bake-in-progress and held clusters stay resident (see [hot-tile residency](DECISIONS.md#2026-06-05-frontend-canvas-hot-tile-residency-frees-covered-cold-clusters)). End-to-end smoke verification at full 1M scale is covered by the `qa-and-load` soak test
- [x] Per-tile loading indicator: a pulsing badge over viewport cells whose content is not yet on screen, the cell being the unit (`loadingOverlay.ts` owns the badge sprites and pulse; the stage drives them each frame from `computeLoadingCells`). Three states, by zoom band: a baked tile not ready at a zoom-out, a group still fetching its AVIF textures at a zoom-in, and a region not yet streamed in. The last case is covered by an additive `coverage` rect on `region_state`: the server now acks the entered viewport area even when it holds no groups, so the client's `knownCells` tells a pending region from a genuinely empty one. The ack is a world rect (not cell keys); the broadcast, cap and LOD grids are now one shared world grid (`WORLD_TILE_SIZE`), so the rect maps cleanly onto the client's LOD cells. The full-board spectator never sets `coverageSeen`, so the not-yet-streamed badge applies only to viewport-streamed contributors
- [x] Per-tile piece cap: a non-merging drop onto a cell already holding `MPP_TILE_PIECE_CAP_MULTIPLIER` (default 8) times the cell's solved density (`(WORLD_TILE_SIZE / pieceSize)` squared) is rejected by the server (`handleDrop` sums the destination cell's group sizes from the `GroupIndex`), so a zoom-out LOD tile never bakes an unbounded pile. The cell is the shared world grid tile (`WORLD_TILE_SIZE`), so the capped region is exactly the LOD tile the client sees and the red flash highlights. The rejected cluster bounces back via the (previously unemitted) `rollback` message: the dropper gets `reason: "tile_full"` and flashes the tile red plus a "Too many pieces on this tile" toast, the neighbours who saw the drag get a plain position correction. Merges and frame anchors are exempt. See DECISIONS: per-tile piece cap on non-merging drops
- [ ] Event-start cascade entrance: synchronized across clients at `eventStartsAt`, pieces fall into their shuffled positions, late joiners skip it

### `backend-realtime`
- [x] Viewport sharding for broadcasts: spatial broadcast index + cluster-AABB scoping
- [x] Viewport sharding for broadcasts: group index + partial-state resync on pan. A second in-process spatial index (`GroupIndex`) on the broadcast cell grid files every group in the cell of its body top-left and stores its position; it is maintained on every committed move (non-merging drop, merge, init/reset/force-complete) and rebuilt from Redis at boot. `handleViewport` resyncs the groups in a client's newly entered cells (already-held cells are not re-sent) via a new `region_state` message (PROTOCOL_VERSION 2), so a peer panning into a region picks up the scoped, non-persisted non-merging drops it missed. The client applies positions behind an ordering guard (`resyncShouldApply`) that skips groups it is holding, a peer is holding, or it just dropped and is awaiting confirmation of, so a resync never rewinds a newer live update. This is the incremental-subscriptions read path, generalized by the viewport-scoped join below into a construction stream
- [x] Viewport sharding: viewport-scoped initial state on join (drop full-state-on-join). `welcome` carries no board (PROTOCOL_VERSION 3); the contributor builds an empty board and streams groups in per viewport via a `region_state` construction stream (origin, locked, size, pieceIds) served from the `GroupIndex`, which now stores the origin (keyed by body-min). A default fit viewport is a global subscription and streams nothing by design (the empty-board-plus-minimap landing), filling in as the user zooms/pans under `MPP_BROADCAST_MAX_CELLS`. `build()`'s whole-board pass is kept for the spectator via a shared `constructGroup`; the stage is partial-board-safe (`applySnap` reconciles membership across the visited-region boundary with no phantom groups, `applyRegionState` upserts construction). The minimap renders from a server-computed density grid (`minimapGrid` on the keyframe, a periodic `minimap` WS message for contributors) plus a live overlay of the locally known regions. Verified at the alpha's 10 000
- [x] Spectator stream scales to 1M: a CDN-cached keyframe (regenerated only while the event is live) plus immutable wall-clock event-log windows (`GET /keyframe` + `GET /events/<t0>`, retiring `GET /snapshot`). The client loads the keyframe, tails windows a few seconds behind live, interpolates group positions between drops and replays snaps in order, and re-bases onto fresh keyframes with no jump, so the per-tick payload is proportional to the events in the interval, not the piece count. Absorbs the snapshot cadence tuning (keyframe interval + idle gate)
- [x] Anti-abuse for public traffic beyond the Phase 1 per-IP rate limit: the anonymous spectator stream (`GET /keyframe` + `GET /events/*`), previously unlimited, now sits behind a `makeSpectatorGuard` boundary that counts every non-preflight request against a Redis per-IP fixed window (`MPP_SPECTATOR_RATE_MAX`/`MPP_SPECTATOR_RATE_WINDOW_SEC`, default 120/60s, fail-open) and rejects any query string (the cache-bust bypass: the edge keys by full URL, so a random `?cb=` would miss the CDN and hit the origin every time; the client never sends one). Rejections carry wildcard CORS + `no-store` so the edge never caches a 400/429 and browser fetches can read the status. Per-account gameplay limits and IPv6 `/64` bucketing stay out of scope

## Backlog

Ideas worth keeping but not yet committed to a phase. Promote into a phase track when scope and timing are clear.

- **Dynamic max-zoom that grows with progress.** Cap zoom-out level early in the puzzle and relax it as more pieces are placed, to bound the visible piece count in any viewport. Lighter alternative or complement to the Phase 2 LOD aggregated tiles. A fixed 15% zoom floor already exists (see [play-zone hard limits](DECISIONS.md#2026-05-21-frontend-canvas-play-zone-hard-limits)); the dynamic, progress-relative version is the open idea here.
- **Coordinate HUD overlay.** Small overlay showing the current viewport position (XY, sector, zoom). Zoom is already shown in the zoom rail, so the marginal value is the XY and a sector readout, which is low at alpha scale and grows with the canvas. Needs a "sector" concept defined first. Revisit at Phase 2 (1M pieces) when orientation on the canvas becomes a real problem.
- **Jump-to search bar (`âŒ˜K`).** Command palette to recenter the camera on a coordinate, sector, or piece id. Depends on a sector concept and a `panTo(worldX, worldY)` camera method (today only `fit` and `center` exist). A navigation aid sized for a huge canvas, not worth building at alpha scale.
- **Bookmark tray (piece shortlist).** Client-side panel of thumbnails for pieces a player flags while hunting; clicking a slot flies the camera to the piece's current position. Geometry is deterministic from the seed, so thumbnails render with no extra data. Pure client-side, no protocol change. No value at alpha scale (the whole board fits on screen); it pays off once the canvas is large enough that pieces cannot be re-found by eye.
